## Context

当前仓库的权限链路已经形成两个稳定但容易混淆的来源：

- `server/service/menu` 基于 `Roles.Menus` 组装 `userinfo.menus` 与 `userinfo.permissions`
- `server/service/role` 基于 `Roles.Apis` 维护 `sys_role_api`，再同步到 `casbin_rule`
- `server/middleware/casbin.go` 继续按 `role_code + path + method` 执行接口鉴权
- `web/src/views/admin/system/role/components/RolePermissionDrawer.vue` 继续提供“角色菜单”和“角色API”两个授权入口

这说明仓库已经不是“一个权限源控制一切”的模型，而是明确的双轨模型。问题不在于模型本身不可行，而在于：

- 用户对模型的心智不稳定
- 角色回显或联动缺口会被误判为权限保存失败
- 权限保存后如果缓存或 Casbin 策略刷新不及时，会造成运行时与数据库不一致

因此本次设计目标不是改变架构，而是固定并澄清当前架构的边界。

约束：

- 继续使用 Gin + GORM + Casbin + JWT 结构
- 不新增权限引擎，不引入统一权限码改造
- 继续沿用后台 Drawer 交互和当前本地 `components/` 拆分
- 不改部门数据范围和用户管理范围控制模型

## Goals / Non-Goals

**Goals:**

- 明确菜单权限与 API 权限的职责边界，并让角色分配界面表达清楚
- 固定角色权限抽屉重新打开时的真实回显行为，减少“未写入数据库”的误判
- 固定按钮型菜单授权时的父级菜单保留逻辑，保证 `userinfo.menus` 与动态路由可用
- 固定角色 API 权限保存后的 Casbin 立即同步行为，避免运行时接口权限滞后
- 用 OpenSpec 为后续权限相关改动建立可验证基线

**Non-Goals:**

- 不把双轨模型切成统一权限码模型
- 不取消 `sys_role_api`
- 不新增菜单与 API 之间的自动映射关系表
- 不修改业务模块权限设计，只关注后台系统角色授权链路

## Decisions

### 1. 保持双轨模型，不做自动互相授权

决策：

- 菜单权限与 API 权限继续分别授权、分别保存、分别生效
- 角色权限页面只允许展示职责说明，不允许隐式把菜单勾选自动变成 API 授权

原因：

- 当前仓库的真实执行链路就是双轨
- 自动联动会让排错路径再次变得不透明
- 既然用户已明确要求“Casbin 继续做后端权限管理”，就不应该在本轮悄悄把授权逻辑合并

备选方案：

- 菜单自动关联 API：用户体验更像单轨，但会引入一套新的隐式映射规则，与当前技术决策冲突

### 2. 菜单权限继续作为 `userinfo` 唯一前端权限来源

决策：

- `userinfo.menus` 和 `userinfo.permissions` 继续只从 `Roles.Menus` 推导
- API 权限不进入 `userinfo.permissions`

原因：

- 当前前端 `v-permission`、动态菜单、按钮显隐都已经围绕菜单权限实现
- 如果把 API 权限混入 `userinfo.permissions`，会造成 UI 与角色菜单树失去一致性

备选方案：

- 让 API 权限也出现在 `userinfo.permissions`：会模糊“前端权限来自菜单”的边界，本轮不采用

### 3. 按钮型菜单授权必须保留祖先链

决策：

- 当角色只勾选按钮或子菜单时，保存层必须同时保留所需父级菜单链
- 用户登录后返回的菜单树也必须能从角色菜单集合中恢复完整可见路径

原因：

- 否则前端会出现“按钮权限存在，但页面菜单树为空或无入口”的坏状态
- 这是当前仓库双轨模型下最关键的菜单可用性保障之一

备选方案：

- 只保存最小按钮节点：实现简单，但会破坏菜单树与动态路由可用性

### 4. 角色 API 权限保存后立即同步 Casbin

决策：

- 角色 API 绑定更新后，服务层必须刷新该角色对应的 Casbin 策略并持久化
- 不允许依赖重启、手工同步或定时任务让接口权限最终一致

原因：

- 当前仓库的后端权限执行层是 Casbin 运行时策略
- 如果数据库与 Casbin 运行时不一致，管理员会直接感知为“明明保存了，为什么接口还是 403”

备选方案：

- 只写 `sys_role_api`，延迟刷新 Casbin：不可接受，会制造明显运行时偏差

### 5. 角色权限界面必须直接表达双轨语义

决策：

- 在“角色菜单”和“角色API”授权区域分别增加简短说明
- 说明必须直接表达：
  - 菜单权限只影响前端显示
  - API 权限只影响后端接口访问
  - 两者不会互相自动授予

原因：

- 这是最小但最高收益的交互改进
- 能降低管理员对“为什么按钮看得见但接口403”或“为什么接口通了但没按钮”的困惑

备选方案：

- 只靠文档说明，不改界面：管理员在实际操作现场仍然容易误判

## Risks / Trade-offs

- [双轨模型仍然存在学习成本] → 接受该成本，但通过界面说明和 spec 固定行为减少误判
- [未来若想改统一权限码，会与本 change 方向不同] → 本变更明确只服务当前决策，不阻止后续单独立项重构
- [角色权限页面文案过多影响可读性] → 说明保持 1 到 2 行，不引入新的复杂说明面板
- [继续分开保存两个权限源] → 是当前用户明确要求，优先于“更统一”的默认偏好

## Migration Plan

1. 为当前双轨模型补齐 OpenSpec 能力描述，固定行为边界
2. 审核并对齐角色权限后端行为：
   - 角色详情回显
   - 菜单祖先链保留
   - Casbin 立即同步
3. 审核并对齐前端角色权限抽屉表达：
   - 菜单/API 分离说明
   - 已有权限信息展示
4. 跑后端与前端验证，确认权限保存、重新打开回显、接口即时生效三条链路成立
5. 如需回滚，仅回滚本次双轨优化相关说明与行为修复，不涉及表结构回滚

## Open Questions

- None. Current technical direction is fixed for this change: Casbin remains the backend authority, and menu/API permissions remain separate.
