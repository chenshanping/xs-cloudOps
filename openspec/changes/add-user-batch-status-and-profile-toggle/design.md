## Context

当前仓库的用户管理已经具备：

- 单个用户状态切换
- 批量删除
- 强制下线
- 基于 JWT + Redis 的 token 白名单/黑名单机制
- 统一的操作日志中间件
- 基于 `sys_config` 的系统参数读写

本次改动同时触达 `server/`、`web/`、权限元数据、系统配置和操作日志审计面，因此需要在编码前固定技术边界，避免把“批量状态修改”和“身份功能”误扩展成更大范围的用户体系重构。

约束：

- 继续沿用现有 Gin + GORM + Casbin + Redis 结构，不引入新依赖
- 后台页面保持现有 `ProTable` + 工具栏模式
- “身份”按钮只做后台用户管理列表中的显示开关，不调整前台身份认证页面

## Goals / Non-Goals

**Goals:**

- 为后台用户管理新增批量启用、批量禁用闭环
- 在后端统一处理批量状态安全校验、缓存清理和 token 失效
- 让后台“身份”按钮可通过系统参数设置控制显示，默认隐藏
- 复用现有权限和操作日志基础设施，而不是另起一套实现

**Non-Goals:**

- 不重构单个状态切换接口
- 不改动前台个人中心、身份注册表或身份档案字段
- 不实现更复杂的角色等级体系；本轮只做与当前仓库模式一致的受保护账号拦截
- 不对操作日志表结构做新增字段扩展

## Decisions

### 1. 使用统一批量状态接口，而不是拆成两个接口

决策：

- 新增一个批量状态接口，前端分别通过“批量启用”和“批量禁用”按钮传不同 `status`

原因：

- 当前单用户状态切换已经使用 `status` 语义，批量接口沿用相同模型最一致
- 避免出现 `/enable-batch`、`/disable-batch` 两套重复路由和重复校验

备选方案：

- 两个独立接口：语义更直白，但会复制参数校验、事务和 token 失效逻辑

### 2. 操作者身份只从 JWT 中间件上下文获取

决策：

- 后端服务逻辑使用 `middleware.GetUserID(c)` 作为当前操作者来源

原因：

- 当前仓库已经在 JWT 中间件中把 `user_id`、`username`、`role_codes` 写入上下文
- 这能保证安全校验和操作日志中的操作者身份不可被前端伪造

备选方案：

- 前端传 `operator_id`：不安全，直接排除

### 3. 批量禁用时复用现有 token 失效能力

决策：

- 当目标状态为禁用时，对成功更新的用户逐个调用现有 `RemoveUserToken`
- 批量启用不主动签发 token，只恢复后续正常登录能力

原因：

- 当前仓库已经使用 Redis 维护用户当前有效 token
- 禁用后仅更新数据库状态不够，必须让已登录用户即时失效
- 复用现有强制下线链路最小、最稳

备选方案：

- 只改数据库状态：实现简单，但被禁用用户可能继续持有旧 token，不能接受

### 4. 受保护用户采用仓库现有“管理员硬保护”思路

决策：

- 后端服务层拒绝：
  - 当前操作者本人出现在“批量禁用”目标中
  - 受保护管理员账号出现在批量状态修改目标中

原因：

- 当前仓库没有完整的角色等级比较体系
- 现有代码已经使用 `admin` / `super_admin` / `ID=1` 一类硬保护思路，先保持局部一致

备选方案：

- 新建完整角色权重体系：长期更好，但超出本轮范围

### 5. “身份”按钮显示控制走独立系统配置项

决策：

- 新增一个独立配置键，例如 `user_profile_button_visible`
- 默认值为 `false`
- 参数设置页维护该开关，用户管理列表按配置决定是否渲染按钮

原因：

- 当前系统配置读取链路已经成熟
- 这是后台用户管理页的局部显示控制，不应复用 `front_mode` 这类前台模式配置

备选方案：

- 复用 `front_mode`：语义不准确，后续维护容易混淆

### 6. 操作日志复用现有中间件，不新增专门日志表写入

决策：

- 不手工插入业务日志
- 让批量状态接口通过现有 `OperationLog` 记录请求体中的 `ids` 和 `status`

原因：

- 当前中间件已经记录：
  - 操作者 `user_id` / `username`
  - 请求路径
  - 请求体
  - 响应状态
- 本次需求的审计关键数据已经足够覆盖

备选方案：

- 单独写业务审计详情：更细，但当前没有强制需求，先不扩表和扩链路

## Risks / Trade-offs

- [受保护账号识别不够细] → 先使用仓库现有管理员保护模式，后续若出现更复杂角色层级，再单独立项
- [批量操作部分成功 vs 全部失败的语义] → 本轮统一为“发现非法目标即拒绝本次提交”，避免前端难以解释半成功状态
- [配置项新增但旧环境无默认值] → 在系统配置默认值初始化和前端默认配置中同时补齐
- [前端仍保留旧多选删除交互] → 批量启用/禁用与批量删除共存，但各自独立确认文案和权限点

## Migration Plan

1. 新增系统配置默认值，保证旧环境没有该键时也能默认隐藏“身份”按钮
2. 发布后端批量状态接口和前端按钮
3. 默认保持“身份”按钮隐藏，不影响现有页面主流程
4. 如需回滚：
   - 前端移除批量状态按钮和配置开关
   - 后端移除批量状态路由和服务
   - 新配置键保留但不再消费，或后续清理

## Open Questions

- None. Current scope and defaults are confirmed: batch enable/disable is included, and the profile button defaults to hidden.
