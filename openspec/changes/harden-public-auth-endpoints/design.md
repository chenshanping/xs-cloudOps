## Context

当前认证链路使用：

- Gin handler 作为公开接口入口
- Redis 存验证码、登录锁定、JWT 白名单/黑名单
- JWT 模型为单用户当前有效 Token，刷新接口允许过期后一段时间内续签

现有问题集中在“公开接口没有统一安全基线”，而不是业务流程本身缺失。

## Decisions

### D1：滑动验证码做安全降级，不继续维持当前实现

当前 `GET /api/v1/captcha/slider` 必须返回 `target_x` 才能让前端自己绘制拼图。这种实现天然把答案暴露给调用方。

本次不尝试在当前任务里重写成完整的服务端拼图图层生成方案，而是采取更小且安全的处理：

- 当系统配置的登录验证码类型为 `slider` 时，后端对外返回 `digit`
- `GET /captcha/slider` 直接返回“不支持”或等价拒绝，不再提供可用挑战

原因：

- 立即消除答案泄露
- 不需要在当前需求中引入复杂图像生成与前端渲染重构
- 登录页已支持普通图形验证码，存在可回退路径

### D2：公开认证接口限流使用 Redis 固定窗口

限流采用 Redis `INCR + EXPIRE` 固定窗口，不引入新依赖。原因：

- 当前仓库已广泛使用 Redis
- 公开认证接口数量有限，固定窗口足够
- 便于按接口和身份维度做最小定制

维度策略：

- 注册：IP + 用户名 + 邮箱
- 发送邮箱验证码：IP + 邮箱
- 用户名重置密码：IP + 用户名
- 邮箱重置密码：IP + 邮箱
- Token 刷新：IP + Token 摘要

### D3：邮箱找回密码对外返回统一成功文案

当邮箱验证码已通过，但系统中不存在该邮箱账号时：

- 不返回 `该邮箱未注册`
- 对外统一返回 `密码重置成功`
- 实际上不执行任何账号写入

原因：

- 这条接口属于密码恢复链路，应避免外部据此确认账号存在性

### D4：JWT 白名单 TTL 覆盖刷新窗口

要想在刷新时“无条件校验白名单一致性”，白名单本身就必须在 refresh window 内仍然存在。

因此：

- `GenerateToken()` 写入白名单时，TTL = `Expires + RefreshWindow`
- `RefreshToken()` 刷新前先检查黑名单，再无条件验证白名单 token 是否仍与当前 token 一致

这样既能保留“过期后一段时间可刷新”的产品行为，也不会放过已被替换掉的旧 token。

## Risks / Trade-offs

- 已配置 slider 的环境会被安全降级成 digit，短期会改变登录交互
- 固定窗口限流不是最精细算法，但实现简单、风险低、足够覆盖公开认证接口
- 邮箱找回统一成功文案会弱化前端对失败原因的区分，但这是安全上更正确的取舍
