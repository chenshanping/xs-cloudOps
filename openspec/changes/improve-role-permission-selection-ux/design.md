## Context

当前角色权限配置已经具备稳定的双轨能力：

- 菜单权限继续通过 `assignMenus` 写入 `sys_role_menu`
- API 权限继续通过 `assignApis` 写入 `sys_role_api` 并同步 Casbin
- 角色详情继续通过 `getRole` 回显 `menus` 与 `apis`

问题不在于权限模型不稳定，而在于当前前端抽屉以两个主 Tab 分开操作。管理员给某个页面配权限时，需要先在菜单 Tab 勾页面和按钮，再切到 API Tab 勾接口，实际操作上下文被切断。

当前约束同样明确：

- 后端没有菜单到 API 的显式映射字段或关系表
- 菜单权限与 API 权限仍必须分开保存
- 不允许把这次交互优化扩成自动联动或统一权限体系重构
- 角色权限仍然使用 Drawer，不改为 Modal 或新页面

## Goals / Non-Goals

**Goals:**

- 在一个同屏 Drawer 中完成页面级角色权限配置，减少菜单/API 来回切换
- 保持菜单权限和 API 权限的分离选择、分离保存、分离生效
- 基于现有 `getMenuTree()`、`getAllApis()` 和 `getRole()` 组装聚合视图，不新增后端接口
- 保持现有祖先链归一化与角色真实回显逻辑
- 单一保存按钮下明确区分菜单保存失败、API 保存失败或部分失败

**Non-Goals:**

- 不新增后端菜单-API 显式映射关系
- 不根据勾选自动推导或同步另一侧权限
- 不修改 Casbin 鉴权模型
- 不新增数据库字段、关系表或升级脚本
- 不修正与本次目标无关的全局权限设计问题

## Decisions

### 1. 权限抽屉改为单视图聚合，而不是保留双 Tab

决策：

- 去掉当前“角色菜单 / 角色API”两个主操作 Tab
- 改为左侧一级菜单导航、右侧页面权限块列表的单视图结构

原因：

- 当前最大痛点是切换成本，而不是缺少信息
- 继续保留双 Tab 再做跳转优化，仍然无法做到同屏对照
- 单视图更适合保持一个统一保存动作

备选方案：

- 保留双 Tab + 快捷定位：改动更小，但不足以解决主要痛点

### 2. 页面聚合只在前端做，不新增后端映射

决策：

- 使用前端聚合视图模型，将菜单树和 API 列表组织成页面权限块
- 不新增 `menu_id`、映射表或聚合接口

原因：

- 当前 `sys_api` 只有 `group/path/method/description`，没有菜单关联字段
- 用户已明确要求本轮不要扩大到后端权限结构改造
- 这轮目标是交互优化，前端编排足够实现

备选方案：

- 新增后端映射元数据：准确性更高，但会把范围扩大到模型与接口设计

### 3. 页面聚合规则固定为“展示定位规则”，不是授权规则

决策：

- API 归类优先按 `group` 匹配页面菜单名
- 次优先按 `path` 中的模块语义匹配
- 再按 `description` 辅助匹配
- 无法稳定匹配的 API 进入“未归类 API”区
- 归类结果只影响展示，不影响保存语义

原因：

- 当前没有后端真映射，前端必须承认“有些 API 只能近似归类”
- 若把归类规则误当成授权规则，就会变成隐式联动，违反范围

备选方案：

- 强制所有 API 都必须归入某个页面：会引入大量错误猜测

### 4. 单一保存按钮继续分两次提交，但使用可区分结果的保存策略

决策：

- 点击保存后仍调用现有两个接口
- 使用 `Promise.allSettled` 或等效分离结果处理，而不是裸 `Promise.all`
- 两边都成功才关闭 Drawer
- 部分失败时保留当前勾选状态，不自动回滚，不关闭 Drawer，并明确提示哪一侧失败

原因：

- 现有后端接口已经稳定，不需要为了单按钮再造聚合保存接口
- 裸 `Promise.all` 无法表达部分失败
- 保留现场状态更有利于管理员补救重试

备选方案：

- 新增后端事务型聚合保存接口：原子性更强，但超出当前边界

### 5. 继续保留角色真实回显和菜单祖先链归一化

决策：

- 打开 Drawer 时继续先拉菜单树、API 列表、角色详情
- 菜单回显继续沿用现有祖先链归一化逻辑
- API 回显继续直接以角色已绑定 `apis` 为准

原因：

- 聚合视图只是重新组织显示，不应改变权限真实来源
- 当前回显链路已经过验证，优先复用

备选方案：

- 改成新聚合数据源：风险高且没有必要

## Risks / Trade-offs

- [API 归类存在近似匹配误差] → 提供“未归类 API”兜底区，不强制错误归类
- [单一保存存在部分失败中间态] → 使用分离结果提示并保留 Drawer 状态，允许直接重试
- [单视图信息密度更高] → 用左侧一级菜单过滤右侧页面块，并保留搜索
- [页面块过多导致滚动负担] → 当前一级菜单点击后只展示该一级菜单下页面块，不一次性铺满全部内容

## Migration Plan

1. 新建 OpenSpec capability，锁定“交互优化、不改双轨模型”的边界
2. 重构角色权限 Drawer 为单视图聚合结构
3. 增加前端页面聚合与未归类 API 兜底逻辑
4. 调整保存逻辑为单按钮双接口、可区分部分失败
5. 跑前端构建与后端回归验证，确认原有双轨能力未受影响

回滚方式：

- 仅回滚前端 Drawer 聚合视图改动
- 保持现有后端接口、角色菜单/API 关系和 Casbin 数据不变

## Open Questions

- None. 本轮采用前端编排、单一保存、未归类 API 兜底，已经足够开始实现。
