## Context

当前仓库的权限语义已经明确分成两条链：

- 菜单/按钮权限用于前端菜单树与按钮可见性
- API 权限用于 Casbin 后端接口放行

这次不是要推翻双轨模型，而是给“双轨之上”的全权限角色增加一个显式、可配置、可迁移的来源。用户希望“角色管理里有个开关”，因此最小稳定实现必须满足：

- 超管身份来源于角色字段，而不是角色编码、角色 ID、用户名或用户 ID
- 一个用户只要拥有任意一个超管角色，就视为拥有全部菜单权限和全部 API 权限
- 普通角色仍按原有菜单/API 分配生效

## Goals / Non-Goals

**Goals**

- 在 `sys_role` 增加显式超管标记
- 角色新增/编辑 Drawer 可配置该标记
- API 鉴权、前端按钮权限、菜单树回显统一使用该标记
- 历史内置 `admin` 角色可通过升级/补齐明确迁移为超管角色
- 保持 Casbin 表结构与角色菜单/API 关联表结构不变

**Non-Goals**

- 不引入用户级超管标记
- 不让 `data_scope` 与超管字段混用
- 不实现“半超管”或仅前端/仅后端超管
- 不重做角色权限抽屉的交互结构

## Decisions

### 1. 使用角色持久化字段表达超管身份

决策：

- 在 `sys_role` 增加布尔字段 `is_super_admin`
- 角色创建、更新、详情、列表响应都返回该字段

原因：

- 用户要的是“角色管理开关”，不是约定俗成的角色名
- 字段语义直接，可审计、可导出、可迁移

备选方案：

- 用角色编码白名单表达超管：仍然是隐式规则，不满足需求
- 用 `data_scope=全部数据` 代表超管：会把数据权限语义和菜单/API 全开耦合，错误

### 2. Casbin 继续做主鉴权，但超管改为显式字段参与

决策：

- 去掉 `rbac_model.conf` 中基于 `r.sub == "admin"` 的特判
- Casbin 中间件仍以角色编码做 `sub`
- 在进入 `Enforce` 前，允许根据当前用户角色集合里的 `is_super_admin` 做统一放行
  或者同步生成一条可审计的超级角色策略，但本轮优先保持最小改动

原因：

- 角色编码仍是 Casbin 当前稳定主体，不需要整体重构
- 显式超管必须与角色实体绑定，而不是再和编码字符串绑定

约束：

- 无论实现放在中间件前判定还是策略同步逻辑中，行为上都必须只由 `is_super_admin` 决定
- 不允许再保留 `admin` 名字后门

### 3. 前端菜单/按钮权限读取也统一走显式超管字段

决策：

- `GetUserPermissions` 只在用户拥有 `is_super_admin=true` 的角色时返回 `["*"]`
- `GetUserMenus` 在用户拥有超管角色时直接返回全部可用菜单树
  若当前实现已经通过角色绑定得到完整菜单，则至少按钮权限全开逻辑必须改成显式字段

原因：

- 现在 API 和前端权限都需要同一套“超管来源”
- 只修 API 不修菜单，会继续出现“接口全开但按钮不一致”或反过来的问题

### 4. 角色表单提供显式开关，但不自动清空已有菜单/API 绑定

决策：

- `RoleFormDrawer.vue` 增加 `a-switch` 或等价开关
- 保存角色时把 `is_super_admin` 与其他基础字段一起提交
- 切换为超管时，不自动删除 `sys_role_menu` / `sys_role_api` 现有绑定

原因：

- 保留原绑定更安全，关闭超管开关后仍能回退到之前的精细权限
- 自动清空会带来不可逆误操作风险

### 5. 历史角色迁移采取“显式补齐”而不是名称继续特判

决策：

- 启动补齐或升级脚本只负责把既定内置超管角色（如当前默认 `admin`）标记为 `is_super_admin=true`
- 迁移完成后，运行时不再依赖 `admin` / `system_admin` 名称做鉴权后门

原因：

- 兼容现有环境，同时把口径收敛到新字段
- 避免升级后行为突变，但又不保留永久魔法值

## Risks / Trade-offs

- [升级环境里历史 admin 角色未被打标] → 需要明确升级脚本/启动补齐策略和验证
- [用户关闭超管开关后权限骤减] → 保留原有菜单/API 绑定，不自动清空
- [API 与菜单链路只修一半] → 本 change 明确要求前后端两条权限链统一读取超管字段
- [Casbin 运行时与数据库角色字段不同步] → 需要测试创建、更新角色后立即生效

## Persistence / Validation

- `sys_role` 新增布尔字段 `is_super_admin`
- 请求校验：
  - create/update role payload 支持 `is_super_admin`
  - 不需要和 `data_scope` 做互斥校验，但实现上超管的有效权限优先级高于普通绑定
- 启动补齐与 SQL 升级：
  - 对 mixed-state 环境保持幂等
  - 仅补缺，不覆盖用户主动修改的非目标字段

## Routing / Service Touch Points

- `server/api/v1/role.go`
  - 角色创建/更新请求支持超管字段
- `server/service/role/role.go`
  - 持久化新字段
- `server/middleware/casbin.go`
  - API 权限放行链路使用显式超管角色
- `server/service/menu/menu.go`
  - 前端权限码与菜单树读取显式超管角色
- `web/src/views/admin/system/role/components/RoleFormDrawer.vue`
  - 表单新增超管开关

## Migration Plan

1. 新增 role super admin capability spec
2. 为 `sys_role` 增加 `is_super_admin`
3. 后端角色请求/响应与服务层支持该字段
4. 移除现有 `admin` 隐式后门，统一改为字段语义
5. 前端角色表单增加开关并回显
6. 运行后端/前端验证，并补回归测试：
   - 超管角色未绑定 API 仍可访问
   - 普通 `admin` 编码角色未绑定 API 时被拒绝
   - 超管角色未绑定按钮权限仍可显示
   - 非超管角色即使 ID=1 或 code=admin 也不能全开
