## Context

当前仓库已有用户、角色、菜单、API、系统配置、动态菜单路由、Casbin 功能权限和基于 JWT/Redis 的登录态能力，但没有部门、组织树、数据范围和统一的数据权限过滤入口。用户管理是本轮首个落地点，因此设计必须同时覆盖 `server/`、`web/`、权限语义、路由注册、数据库迁移与兼容默认值，避免后续再返工底座。

约束：

- 继续沿用 Gin + GORM + Casbin + Redis，不引入新依赖。
- 目录保持 `server/` 和 `web/`，不回到代码生成器工作流。
- 部门管理属于正式内置能力，不做真正可拔插插件。
- 菜单/页面显示允许通过配置开关隐藏，但底层部门与数据权限模型始终存在。
- 前端非平凡弹层默认使用 `Drawer`，并把表单内容拆到本地 `components/`。

## Goals / Non-Goals

**Goals:**

- 建立部门树、用户部门归属、角色数据范围和自定义部门范围的持久化模型。
- 提供统一的数据权限上下文解析与 GORM 过滤入口，先接入用户管理列表与关键用户操作。
- 提供完整的部门管理后台页面和后端接口，包含树形维护、层级校验、删除限制和权限点。
- 为旧环境补齐兼容默认值：已有角色默认保留“全部数据”，已有用户默认归到根部门或平台部门，避免升级后现有能力失效。
- 保留 `dept_module_enabled` 作为展示层开关，只控制菜单与页面是否开放。

**Non-Goals:**

- 不在本轮把全部业务模块都切到部门数据权限。
- 不新增复杂的岗位、人事档案、汇报链、组织编码体系。
- 不改变现有 Casbin 功能权限授权模型的主流程。
- 不把部门管理改成独立插件安装/卸载能力。

## Decisions

### 1. 底层模型正式内置，展示层单独开关

决策：

- `sys_dept`、`sys_role_dept`、`sys_user.dept_id`、`sys_role.data_scope` 永久属于主系统模型。
- `dept_module_enabled` 仅控制部门菜单/页面显隐，不影响表结构和权限计算。

原因：

- 数据权限一旦进入用户、角色和业务查询链路，就不能允许底层模型被抽掉。
- 把“可插拔”限制在展示层，可以兼顾未来按项目隐藏菜单的诉求。

备选方案：

- 真正插件化底层模型：会导致用户、角色、查询逻辑出现大量条件分支，不可控。

### 2. 部门树使用 `parent_id + ancestors` 表达

决策：

- `sys_dept` 至少包含 `id / parent_id / ancestors / name / sort / status / remark`。
- `ancestors` 保存祖先路径，便于高效计算“本部门及下级”。

原因：

- 当前项目尚无树形基础设施；`ancestors` 能让查询和删除校验比纯递归更稳定。

备选方案：

- 仅使用 `parent_id` 递归查下级：实现可行，但后续数据权限过滤和批量校验成本更高。

### 3. 角色数据范围采用固定枚举，并支持自定义部门

决策：

- `data_scope` 使用稳定枚举：
  - `1` 全部数据
  - `2` 自定义部门
  - `3` 本部门
  - `4` 本部门及下级
  - `5` 仅本人
- 角色与部门的自定义范围通过 `sys_role_dept` 关联维护。

原因：

- 这是后台系统最常见且可解释的五类范围，足够支撑当前与后续扩展。
- 自定义部门不能塞进角色备注或配置 JSON，必须结构化存储。

备选方案：

- 只支持全部/本人：太弱，无法承接部门权限底座。
- 用独立 JSON 配置表：前后端和查询层都更难约束。

### 4. 数据权限统一从操作者角色集合解析，再以通用过滤入口应用

决策：

- 新增统一的数据权限服务，负责根据当前用户的角色集合解析最终生效的数据范围。
- 提供统一的 GORM 过滤入口，用于把“部门列 + 本人列”映射到查询条件。
- 本轮先接入用户管理：列表、详情、创建、编辑、删除、批量删除、状态修改、批量状态、重置密码、强制下线、身份查看等用户管理接口。

原因：

- 数据权限必须保护直接接口调用，不能只依赖前端按钮显隐。
- 统一入口能避免后续各业务自己散写 `WHERE dept_id ...`。

备选方案：

- 每个 service 自己写过滤逻辑：很快会失控，且难以复查安全边界。

### 5. 兼容默认值采用“平台根部门 + 角色默认全部数据”

决策：

- 初始化和升级时补齐一个根部门，例如“平台”。
- 旧用户若 `dept_id` 为空，升级脚本和初始化逻辑回填到根部门。
- 旧角色若 `data_scope` 为空或缺失，默认回填为“全部数据”。

原因：

- 这样能保持现有系统行为不收窄，避免升级后管理员突然查不到数据。

备选方案：

- 旧用户保持空部门：会让本部门/本部门及下级判断语义不稳定。
- 旧角色默认本人：破坏兼容性。

### 6. 前端按现有系统页模式实现树表和 Drawer 表单

决策：

- 新增 `web/src/views/system/dept/index.vue`，列表主界面使用树形表格。
- 新增本地 `components/DeptFormDrawer.vue` 承载新增/编辑表单。
- 用户页增加部门列和部门选择；角色页增加数据范围选择和“自定义部门”树选择。

原因：

- 保持现有 Ant Design Vue + ProTable + Drawer 的仓库风格一致。
- 复杂表单单独拆组件，避免继续把页面写成巨型单文件。

备选方案：

- 所有表单直接内联在页面：维护成本高，和已确认规则冲突。

### 7. 历史父部门绑定用户采用“保留旧值、禁止新增扩散”的兼容策略

决策：

- 只有叶子部门可以新绑定用户。
- 如果某个部门原本是叶子部门，后续新增了子部门，导致已有用户落在现在的父部门上，这些用户视为历史兼容数据。
- 历史兼容用户不自动迁移；列表、筛选和人数统计继续按当前直属关系展示。
- 编辑历史兼容用户时，若部门未变化，允许保存其他字段；若调整部门，则必须改为叶子部门。

原因：

- 直接强制迁移会把组织调整误当成人员归属调整，存在业务误判风险。
- 若连非部门字段编辑都被拦截，会让历史数据进入“不可维护”状态。

## Risks / Trade-offs

- [用户管理接口接入面较大] → 先抽统一权限检查工具，再逐个接到用户相关 service，避免散点实现。
- [旧环境升级脚本重复执行] → SQL 脚本按幂等方式写，种子/API/菜单用现有 `ensureBuiltInData()` 补齐。
- [部门删除涉及下级和用户归属] → 明确禁止删除存在下级部门或已归属用户的部门。
- [角色选择“自定义部门”时前后端状态不同步] → 角色详情接口直接返回 `data_scope` 和 `dept_ids`，前端按此回填。
- [展示层关闭但接口还在] → 菜单隐藏不等于授权，接口继续受菜单/API 权限和数据权限控制。

## Migration Plan

1. 补齐 OpenSpec 变更并按任务实现后端模型、迁移、权限过滤与部门接口。
2. 增加初始化与升级兼容逻辑：根部门、角色默认数据范围、旧用户部门回填、菜单/API/配置补齐。
3. 上线部门管理页面、用户部门字段、角色数据范围配置，并默认关闭部门菜单展示开关。
4. 验证用户管理数据权限链路后，再逐步把统一过滤入口推广到其他业务模块。
5. 如需回滚：
   - 先关闭菜单/页面展示
   - 回滚前后端接口暴露与用户管理接入
   - 数据表和字段按升级脚本另行评估，不直接做破坏性删除

## Open Questions

- None. 当前已确认：部门管理按权限底座正式写入，可插拔只保留在展示层开关，不保留在底层模型层。
