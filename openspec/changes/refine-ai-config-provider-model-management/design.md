## Context

当前仓库的 AI 配置数据已经以 `ai_config` JSON 的形式保存在 `sys_config` 中，前端也已有基本的加载、编辑和保存能力。但现有页面把平台字段和模型列表混在一个折叠区里，不适合继续承载“平台切换、平台字段维护、远端模型发现、按需导入、统一保存”这几个不同层次的管理动作。

本次改动同时触达 `server/`、`web/`、AI 模块路由与启动补齐逻辑，因此需要先固定 UI 组织方式、远端模型发现接口、导入语义和权限补齐边界，避免把一次配置页重构扩大成 AI 对话链路重构。

约束：

- 继续沿用现有 `ai_config` JSON 落库，不新增表、不改 SQL
- 继续沿用 Gin + GORM + Casbin + JWT + Vue 3 + Ant Design Vue 结构
- 非 trivial 弹出交互默认使用 Drawer，并把 Drawer 内容拆到局部 `components/`
- 新增远端模型发现必须由后端代理，不能前端直连第三方

## Goals / Non-Goals

**Goals:**

- 把 AI 配置页重构为左侧平台列表、右侧模型主面板的方案 B
- 为平台信息编辑提供独立 Drawer
- 为平台模型发现与导入提供独立 Drawer
- 新增后端代理接口，使用当前编辑态的 `api_key` 和 `base_url` 拉取平台模型列表
- 把导入语义固定为“追加去重 + 仅回填编辑态 + 统一保存落库”
- 补齐新接口权限和启动补齐逻辑，并保证不覆盖管理员已修改菜单元数据

**Non-Goals:**

- 不改动 AI 聊天主页面、对话存储和模型调用流程
- 不实现自动同步、定时刷新、远端模型缓存或导入历史
- 不拆分 `ai_config` 为多张表或多条配置记录
- 不做更复杂的供应商适配层；本轮只支持 OpenAI 兼容模型列表拉取

## Decisions

### 1. 页面采用“左列表 + 右模型主面板 + 两个独立 Drawer”

决策：

- 主页面左侧显示平台列表
- 主页面右侧只聚焦当前平台的已导入模型
- 平台字段编辑和平台模型管理使用两个独立 Drawer

原因：

- 这是用户已经确认的交互方向
- 平台字段维护和模型导入是两类不同任务，拆开后更清晰
- 符合仓库对非 trivial 弹出交互优先使用 Drawer 的既有约束

备选方案：

- 单一大表单：当前问题来源，直接排除
- 一个 Drawer 内双页签：可行，但状态耦合更高，切页和保存更复杂

### 2. 远端模型拉取使用当前编辑态表单值，而不是要求先保存

决策：

- 平台模型管理 Drawer 打开时，优先读取当前页面内该平台的 `api_key` 和 `base_url`
- 若当前值不完整，则禁止拉取并提示补全

原因：

- 用户已确认不希望为了试拉模型先强制保存平台配置
- 这样能支持“改 Key / 改 Base URL -> 立刻试拉模型 -> 再决定是否保存”

备选方案：

- 只允许使用已保存配置：实现简单，但交互笨重，不符合本次目标

### 3. 新增一个后台代理接口统一做模型发现

决策：

- 新增 `POST /api/v1/ai/providers/models/fetch`
- 请求体仅包含 `base_url`、`api_key`
- 返回清洗后的模型列表

原因：

- 避免前端跨域和密钥暴露
- 后端便于集中处理 URL 拼接、超时、日志脱敏、第三方错误规范化

备选方案：

- 前端直连第三方：会暴露密钥且容易被 CORS 拦截，直接排除

### 4. 模型列表拉取按 OpenAI 兼容协议做双路径尝试

决策：

- 若 `base_url` 以 `/v1` 结尾，则请求 `${base_url}/models`
- 否则先尝试 `${base_url}/v1/models`
- 失败后回退尝试 `${base_url}/models`

原因：

- 当前仓库的 AI 对话调用已经按 OpenAI 兼容风格处理 `/chat/completions`
- 实际平台配置中 `base_url` 有的写到域名，有的已写到 `/v1`

备选方案：

- 只拼一种 URL：失败率高，兼容性不够

### 5. 导入语义固定为追加去重，不覆盖本地已编辑模型展示字段

决策：

- 以 `model.id` 作为唯一键
- 远端勾选导入时，只追加不存在的模型
- 已存在模型不重复添加，也不覆盖已有 `name`、`description`

原因：

- 用户已确认导入方式为“追加去重”
- 本地展示名称与描述可能被管理员人工调整，不能被二次导入覆盖

备选方案：

- 全量覆盖：会破坏本地手工调整
- 导入即覆盖部分字段：行为更难预测，本轮不做

### 6. 导入只更新当前编辑态，继续复用现有统一保存心智

决策：

- 点击“导入”后，只把结果合并进当前页面内存状态
- 仍需点击页面底部“保存配置”才真正写入 `sys_config`

原因：

- 用户已确认保存时机为统一保存
- 与当前配置页的脏状态提示、离开拦截和快照恢复逻辑一致

备选方案：

- 导入即落库：会让局部操作绕开整体配置保存流程，破坏一致性

### 7. 新接口权限与菜单补齐沿用现有 AI 模块 repair 模式

决策：

- 新接口纳入 `server/router/modules/ai.go`
- 在 `server/initialize/db_tables.go` 中补齐 API 记录与 admin/system_admin 角色关联
- 使用 create-only / missing-only 风格，不更新已存在记录元数据

原因：

- 当前仓库已经对 AI 菜单和 API 补齐建立了可复用模式
- 仓库明确要求启动补齐不能覆盖管理员已修改内容

## Risks / Trade-offs

- [第三方平台返回结构不完全兼容] → 后端只依赖最小字段 `data[].id`，其余字段尽量容错解析
- [模型列表很多导致前端渲染重] → 管理 Drawer 提供搜索、默认不自动全选，只导入管理员勾选项
- [当前编辑态与数据库已保存态不一致时用户混淆] → 保留现有脏状态提醒，并在导入后明确提示仍需保存配置
- [Base URL 书写不规范导致请求失败] → 后端统一 trim 尾部斜杠并按 `/v1/models`、`/models` 双路径尝试
- [日志泄漏 API Key] → 新接口实现中禁止输出原始请求体或完整 Header，仅返回脱敏错误

## Migration Plan

1. 新增 OpenSpec 变更文档并完成实现
2. 发布后端模型发现接口与前端 Drawer 交互
3. 旧数据继续按原 `ai_config` JSON 读取，无需迁移
4. 如需回滚：
   - 删除新接口与前端模型管理 Drawer
   - 恢复原有 AI 配置页面布局
   - 保留已存在的 `ai_config` 数据，不需要数据回滚
