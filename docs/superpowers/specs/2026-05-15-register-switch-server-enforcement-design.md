# 注册开关后端强制校验设计

## 1. 背景

当前系统配置页提供了 `enable_register` 开关，前端登录页也会根据该配置决定是否展示“立即注册”入口。

现状问题是：即使后台把注册功能关闭，攻击者或普通调用方仍然可以直接调用公开接口 `POST /api/v1/auth/register` 完成注册。这说明当前控制仅停留在前端展示层，没有形成后端安全边界。

本次设计只修复这个问题本身，不扩展到其他认证流程重构。

## 2. 已确认决策

- 目标是修复“关闭注册后，后端接口仍可注册”的问题
- 只做当前问题的最小闭环，不顺手改其他认证逻辑
- 采用方案 A：在 `server/service/user` 中增加后端强制校验
- 前端现有入口显示逻辑保留，不作为本次主要修复点
- 需要补回归测试，证明关闭注册时接口确实失败

## 3. 目标

在后端形成真正生效的注册开关控制，达到以下结果：

- 当 `enable_register=false` 时，任何通过 `POST /api/v1/auth/register` 发起的注册请求都必须失败
- 当 `enable_register=true` 时，现有正常注册流程保持可用
- 校验逻辑位于服务层，避免未来新增其他入口时绕过限制
- 为该行为增加自动化回归测试

## 4. 非目标

以下内容不在本次范围内：

- 不修改登录逻辑
- 不调整忘记密码逻辑
- 不重做系统配置页面
- 不新增新的配置项
- 不扩展为完整认证安全治理改造

## 5. 根因分析

当前公开注册接口位于 `server/api/v1/auth.go` 的 `Register` 方法。

该方法当前仅处理：

- 请求参数绑定
- 注册验证码校验
- 注册邮箱验证码校验
- 调用 `service.User.Register(...)`

但无论在 handler 还是 service 层，都没有针对 `enable_register` 做后端强制判断。

因此当前系统实际行为是：

- 前端：入口可能隐藏
- 后端：公开路由依然接受注册请求

这是典型的“只做前端控制，没有服务端授权边界”的问题。

## 6. 方案比较

### 方案 A：在用户服务层强制校验

在 `server/service/user` 增加“是否允许公开注册”的判断，并在 `Register` 入口执行拦截。

优点：

- 校验位置更接近业务入口
- 后续即使别的 handler、job 或内部调用复用 `User.Register`，也不会绕过
- 改动小，覆盖面足够

缺点：

- 如果未来需要区分“公开注册”和“后台代建用户”，需要继续保持入口语义清晰

结论：

本次采用该方案。

### 方案 B：只在 API handler 校验

在 `server/api/v1/auth.go` 中调用配置并直接拦截。

优点：

- 改动位置直观

缺点：

- 只能覆盖当前 HTTP 入口
- 未来复用 `User.Register` 时容易再次漏掉

结论：

不采用。

### 方案 C：handler 与 service 双重校验

优点：

- 冗余防御更强

缺点：

- 当前需求下存在重复逻辑
- 对本次最小修复来说性价比一般

结论：

暂不采用，优先保持最小闭环。

## 7. 推荐方案

本次采用方案 A，在 `server/service/user` 中新增统一判断，并由 `Register` 方法在创建用户前执行。

建议行为：

- 读取系统配置 `enable_register`
- 当配置不存在、为空或显式为关闭值时，返回明确错误，例如 `系统已关闭注册`
- 仅当配置值为 `true` 或等价开启值时，允许继续注册流程

这样可以把“是否允许公开注册”定义成后端业务规则，而不是前端界面规则。

## 8. 实现边界

建议改动范围限定为：

- `server/service/user/user.go`
- `server/service/user/user_test.go`

必要时允许 `server/api/v1/auth.go` 保持不变，因为本次推荐把规则收口到 service 层。

如果现有测试结构需要，也可以在测试里补充最小的配置数据准备，但不应扩展到别的模块。

## 9. 错误处理

当注册关闭时，`User.Register(...)` 返回明确业务错误，API 层沿用现有 `response.Fail(c, err.Error())` 返回给前端。

本次不引入新的错误码体系，保持与现有仓库错误返回风格一致。

推荐错误文案：

- `系统已关闭注册`

## 10. 测试策略

本次必须增加服务层回归测试，至少覆盖以下场景：

### 10.1 关闭注册时应拒绝创建用户

- 准备 `sys_config.enable_register=false`
- 调用 `User.Register(...)`
- 断言返回错误为 `系统已关闭注册`
- 断言数据库中没有新增对应用户

### 10.2 开启注册时应允许正常注册

- 准备 `sys_config.enable_register=true`
- 调用 `User.Register(...)`
- 断言注册成功
- 断言用户记录已创建

测试重点是证明修复前会失败、修复后会通过，并防止后续回归。

## 11. 风险与注意事项

### 11.1 配置兼容性

现有前端使用的是字符串值 `true/false`。后端实现时应与现有配置存储方式保持一致，至少兼容：

- `true`
- `false`

若仓库其他地方对布尔型配置存在 `1/0` 用法，可以一并兼容，但不应超出本次必要范围。

### 11.2 不影响后台创建用户

本次修复只针对公开注册入口 `User.Register(...)`，不应影响管理员在用户管理页面创建用户的能力，因为那条链路使用的是 `CreateUser(...)`。

### 11.3 不依赖前端是否隐藏入口

即使前端未来某处误显示了注册入口，只要后端校验存在，注册仍然会被拒绝。这正是本次修复要建立的边界。

## 12. 最终结论

这个问题的根因不是“系统配置没保存”，而是“注册开关没有在后端真正执行”。

本次采用最小且稳妥的修复方式：

- 在 `server/service/user` 增加注册开关后端校验
- 在 `User.Register(...)` 中强制执行
- 使用服务层测试覆盖关闭与开启两种场景

这样可以在不扩展需求的前提下，补上公开注册接口缺失的服务端防线。

## 13. 公开接口附加安全检查

在确认注册开关问题后，对当前公开接口做了附加检查，重点覆盖：

- 公开认证接口
- 公开验证码接口
- 公开配置读取接口
- Token 刷新入口

结论是：除了“注册开关未在后端执行”外，当前还存在多处公开接口安全风险，其中包含可直接利用的高危问题。

## 14. 已发现问题

### 14.1 高危：按用户名重置密码接口可绕过验证码校验

涉及接口：

- `POST /api/v1/auth/reset-password-by-username`

涉及代码：

- `server/api/v1/auth.go`

当前逻辑使用：

- 当 `captchaId == ""` 时才去校验验证码

这意味着只要请求里传入任意非空 `captcha_id`，后端就不会真正执行验证码校验，随后直接根据用户名重置密码。

风险：

- 攻击者一旦知道用户名，就可能无需有效验证码直接重置该用户密码
- 这是直接影响账户接管的公开接口漏洞

判断：

- 这是独立于本次注册开关问题之外的高危缺陷
- 优先级应高于一般安全优化项

建议：

- 修正条件判断，改为“验证码参数缺失时拒绝；参数存在时必须校验通过”
- 为该接口增加回归测试，覆盖“伪造非空 captcha_id 但未通过验证时必须失败”

### 14.2 高危：滑动验证码接口直接向前端暴露目标坐标

涉及接口：

- `GET /api/v1/captcha/slider`

涉及代码：

- `server/api/v1/captcha.go`
- `server/service/captcha/captcha.go`

当前接口返回：

- `target_x`

而 `target_x` 正是服务端后续校验滑块是否命中的核心值。公开返回这个字段，相当于把答案直接发给调用方。

风险：

- 调用方无需识别图片，也无需真正做人机验证
- 任何脚本都可以读取 `target_x` 后再调用验证接口完成验证码通过
- 所有依赖该滑动验证码的公开流程都会被削弱

判断：

- 这是验证码实现层面的直接失效

建议：

- `GET /captcha/slider` 不应返回用于判定通过的真实目标坐标
- 若前端需要绘制缺口，应改为由服务端返回已生成好的挑战图层，或返回不可直接还原答案的安全数据
- 增加针对验证码流程的安全回归测试

### 14.3 中高危：多个公开认证接口缺少频率限制与滥用防护

涉及接口：

- `POST /api/v1/auth/register`
- `POST /api/v1/auth/send-email-code`
- `POST /api/v1/auth/reset-password`
- `POST /api/v1/auth/reset-password-by-username`
- `POST /api/v1/auth/reset-password-by-email`
- `POST /api/v1/auth/refresh`
- `POST /api/v1/captcha/slider/verify`

现状：

- 登录接口具备失败计数与锁定逻辑
- 但注册、邮箱验证码发送、密码重置、刷新 Token、滑动验证码验证等公开入口，没有看到同等级别的限流或频率控制
- `send-email-code` 的验证码校验也是“可选”的，不是强制前置条件

风险：

- 可被批量刷注册、刷验证码、撞库式尝试、邮件轰炸、Token 刷新滥用
- 即使单次请求逻辑正确，缺少速率控制后仍然容易被自动化攻击放大

外部安全基线参考：

- OWASP API Security Top 10 2023 指出凭证恢复类接口应按登录接口同等级别对待，需做暴力尝试与频率控制
- OWASP Forgot Password Cheat Sheet 建议对找回密码与验证码发送加入限流、验证码或其他防自动化措施

建议：

- 为公开认证接口增加按 IP、按账号标识、按邮箱维度的限流
- 对验证码发送接口增加短周期发送冷却与日级配额
- 把“是否要求图形验证码”从前端约定升级为后端强制策略

### 14.4 中危：邮箱找回密码流程存在账号枚举风险

涉及接口：

- `POST /api/v1/auth/reset-password-by-email`

现状：

- 邮箱验证码校验通过后，后端会继续查询该邮箱对应用户
- 若不存在，返回 `该邮箱未注册`

风险：

- 攻击者可借助该接口或相邻流程推断某邮箱是否为系统有效账户
- 这会给后续撞库、社工或定向攻击提供前置信息

外部安全基线参考：

- OWASP Forgot Password Cheat Sheet 建议对存在与不存在账户返回一致响应，避免枚举

建议：

- 对外返回统一文案，避免暴露“邮箱是否已注册”
- 内部日志可保留真实原因，前端提示保持模糊一致

### 14.5 中危：Token 刷新入口对白名单校验不完整

涉及接口：

- `POST /api/v1/auth/refresh`

涉及代码：

- `server/api/v1/auth.go`
- `server/utils/jwt.go`

当前 `RefreshToken` 逻辑中，只有在旧 Token 已进入黑名单时，才额外检查它是否仍然是 Redis 白名单中的当前有效 Token。若旧 Token 未进黑名单，则可能跳过对白名单的一致性校验。

这会削弱当前仓库“Redis 白名单 + 单用户当前有效 Token”的会话模型。

风险：

- 某些已被替换但仍在刷新窗口内的旧签名 Token，可能仍有机会换发新 Token
- 这会弱化“异地登录挤掉旧会话”或“只认当前 Token”的预期

建议：

- 刷新前应无条件校验该 Token 是否仍然匹配当前白名单
- 白名单不匹配时直接拒绝刷新
- 为“旧 Token 被替换后不可刷新”补测试

## 15. 与本次修复范围的关系

当前文档已同时记录：

- 本次已确认要修复的问题：关闭注册后后端仍可注册
- 额外排查发现的公开接口安全问题

其中：

- `reset-password-by-username` 验证码绕过
- `captcha/slider` 直接暴露答案

都属于高优先级问题，应尽快单独修复，或在用户同意后纳入下一轮处理。

本轮若严格按已确认范围执行，则代码实现仍先聚焦“注册开关后端强制校验”；其余问题先保留为已确认风险记录。
