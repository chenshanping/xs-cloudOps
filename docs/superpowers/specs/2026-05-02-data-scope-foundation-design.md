# 数据权限基础框架设计

日期：2026-05-02

## 1. 背景

当前项目已经具备以下数据权限基础能力：

- 角色默认数据范围：`sys_role.data_scope`
- 模块级数据范围覆盖：`sys_role_data_scope`
- 数据权限资源码注册：`server/service/core/data_scope_resource.go`
- 查询期数据范围解析与注入：`server/service/core/data_scope.go`
- 已验证接入模块：用户管理、部门管理

本次设计不再讨论“是否需要数据权限”，而是将现有能力明确为项目级基础框架标准，供后续业务模块按统一规则快速接入。

## 2. 目标

将当前数据权限能力正式定义为“基础框架”，满足以下目标：

- 后续业务模块可以按统一规范快速接入数据权限
- 不需要每个业务模块重新设计“本人 / 本部门 / 本部门及以下 / 自定义部门”逻辑
- 列表、详情、编辑、删除、批量操作使用同一套资源级数据权限口径
- 保持实现成本可控，不引入过重的规则引擎或独立权限平台

## 3. 定位

本项目的数据权限能力定位为：

- 一个可复用的后台数据权限基础框架
- 优先服务于管理后台 CRUD 型业务模块
- 以组织、部门、创建人为主线
- 采用“角色默认范围 + 模块级覆盖”的企业后台常见模式

本项目当前不定位为：

- 通用 ABAC 平台
- 可视化权限表达式引擎
- 数据库原生行级安全平台
- 一次性覆盖所有未来业务场景的终局设计

## 4. 核心模型

### 4.1 角色默认范围

角色通过 `sys_role.data_scope` 定义默认数据范围。

当前范围枚举固定为 5 档：

- `1 = 全部`
- `2 = 自定义`
- `3 = 本部门`
- `4 = 本部门及以下`
- `5 = 仅本人`

该字段作为所有业务模块的兜底默认值。

### 4.2 模块级覆盖

角色可通过 `sys_role_data_scope` 对某个具体资源模块覆盖默认范围。

关键字段：

- `role_id`
- `resource_code`
- `data_scope`
- `depts`，仅在 `自定义` 范围下使用

规则：

- 若角色在指定 `resource_code` 上存在覆盖配置，则优先使用覆盖值
- 若不存在覆盖配置，则回退到角色默认 `data_scope`

### 4.3 多角色合并

一个用户可能拥有多个角色。模块级数据权限合并规则采用并集模型：

- 任一角色命中 `全部`，该资源模块直接视为全量可见
- 多个角色的部门集合合并
- 多个角色的 `仅本人` 规则合并
- 多个角色的创建人集合合并

不采用“取最严”模型，避免角色叠加后反而缩小权限导致管理体验异常。

## 5. 资源码规范

### 5.1 资源码用途

`resource_code` 表示“数据权限控制的业务模块”，不是菜单编码，也不是 API 编码。

它的职责是描述：

- 这个模块的数据访问范围如何解析
- 这个模块的列表、详情、编辑、删除是否共享同一套数据权限口径

### 5.2 命名规范

推荐格式：

- 系统模块：`system:<module>-management`
- 业务模块：`biz:<module>-management`

示例：

- `system:user-management`
- `system:dept-management`
- `biz:customer-management`
- `biz:order-management`
- `biz:project-management`

要求：

- 同一业务模块只使用一个主资源码
- 列表、详情、编辑、删除、批量操作统一复用这个资源码
- 不允许同一模块的不同操作自行定义多套互相冲突的数据权限资源码

## 6. 数据归属字段规范

后续业务模块若要接入当前数据权限基础框架，数据表应至少具备以下字段之一：

- `dept_id`
- `created_by`

推荐标准：

- 优先同时具备 `dept_id` 和 `created_by`

原因：

- `dept_id` 用于部门维度范围过滤
- `created_by` 用于 `仅本人` 场景
- 两者同时存在时，后续扩展空间最大

不满足该条件的业务模块，不应强行接入当前数据权限框架，应先单独评估数据归属模型。

## 7. 统一行为规则

### 7.1 列表查询

列表查询必须通过统一的资源级数据权限解析后再拼接过滤条件。

标准调用链：

- `ResolveUserDataScopeForResource`
- `Apply...Scope`

不允许在各业务模块中手写一套新的数据范围判断分支。

### 7.2 单条详情

详情查询不能绕过数据权限。

即使用户知道某条数据的 ID，也必须先经过资源级可见性校验，再允许返回。

### 7.3 编辑、删除、批量操作

编辑、删除、批量操作必须和列表查询共享同一个 `resource_code` 口径。

要求：

- 列表中能看到的数据，才允许继续编辑、删除、批量处理
- 列表中不可见的数据，不能通过直接调用接口绕过

### 7.4 特殊说明：仅本人

`仅本人` 的语义由资源模块定义，但必须保持固定、可预期。

当前框架约定：

- 用户管理：`created_by = 当前用户`
- 不能将“仅本人”解释为 `id = 当前用户` 后再混入后台管理场景

若某个业务模块无法给出稳定的一致语义，则不应接入当前数据权限框架。

## 8. 当前已纳入框架的模块

当前基础框架已明确支持：

- 用户管理
- 部门管理

这两个模块的正常运行，证明以下能力已经成立：

- 角色默认数据范围可用
- 模块级覆盖可用
- 部门树范围解析可用
- 数据权限查询注入可用
- 资源级访问控制已具备基础可复用性

## 9. 后续业务模块接入规范

后续任一业务模块若要接入数据权限，必须按以下步骤执行：

1. 明确业务数据归属字段
2. 定义唯一的 `resource_code`
3. 在核心查询入口接入统一数据权限解析
4. 让详情、编辑、删除、批量操作复用同一资源码
5. 在角色管理中补充该模块的数据权限配置入口
6. 补充至少一组“可见 / 不可见 / 可编辑 / 不可编辑”的回归测试

接入完成的标准不是“列表能过滤”，而是“整个模块的数据访问语义一致”。

## 10. 管理端配置要求

角色管理中的数据权限配置应保持两层结构：

- 角色默认范围
- 模块级覆盖范围

模块级配置建议表现为：

- 一行一个资源模块
- 支持“继承默认值”
- 支持切换到：
  - 全部
  - 自定义
  - 本部门
  - 本部门及以下
  - 仅本人
- 仅当选择“自定义”时展示部门树选择

目标是降低后续业务模块接入后的运维理解成本。

## 11. 不做的范围

本阶段明确不做以下内容：

- 不做通用表达式权限引擎
- 不做 Casbin 承担数据权限计算
- 不做 SQL 片段可视化配置
- 不做数据库级 RLS 方案
- 不为了“更通用”重写当前已有用户/部门数据权限能力
- 不强行让所有业务模块都必须接入当前框架

## 12. 企业化结论

从企业后台开发视角看，当前框架的正确定位是：

- 数据权限基础设施已经建立
- 用户管理、部门管理是首批已验证模块
- 后续业务模块只需遵守统一接入规范即可快速落地

因此，下一阶段的重点不是继续抽象框架，而是：

- 固化资源码规范
- 固化业务接入标准
- 在角色管理端补齐可配置入口
- 让新业务模块按标准逐个接入

## 13. 最终结论

本项目的数据权限标准正式定义为：

- 模型：`角色默认范围 + 模块级覆盖`
- 资源单位：`resource_code`
- 归属字段：`dept_id`、`created_by`
- 合并策略：多角色并集
- 接入标准：列表、详情、编辑、删除、批量统一口径

该方案足以支撑当前项目作为“基础框架”的定位，并为后续业务模块提供稳定、可扩展、成本可控的数据权限接入基础。
