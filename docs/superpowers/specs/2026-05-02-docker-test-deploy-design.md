# Docker 测试环境部署设计

日期：2026-05-02

## 1. 背景

当前项目已经具备完整的前后端业务能力，但测试环境部署方式仍不稳定，存在以下问题：

- 现有 [DEPLOY.md](E:\go_project\go-base\DEPLOY.md) 中的 Docker 方案偏示例化，尚未形成可直接落地的仓库资产
- 前端 `web/.env.test` 仍写死外部测试 IP，换机器或换端口需要重新调整打包口径
- 后端默认只读取固定 `server/config.yaml`，不利于区分本地开发与 Docker 测试环境
- 测试环境需要快速拉起可用闭环，包括 `MySQL`、`Redis`、后端服务、前端页面
- 文件上传目录与日志目录需要持久化，否则容器重建后会丢失排障信息和已上传文件

本次目标不是做完整生产发布平台，而是为当前项目封版前提供一套稳定、可复现、可直接部署到 test 环境的 `docker compose` 方案。

## 2. 目标

本次 Docker 测试环境部署设计需要满足以下目标：

- 使用 `docker compose` 一次性拉起 `mysql`、`redis`、`server`、`web`
- 测试环境通过 `服务器IP + 端口` 访问，不引入域名、HTTPS、额外网关
- 首次启动自动初始化数据库结构和默认基础数据，能直接进入系统登录页
- 前端不再写死外部 API 地址，改为通过容器内 `nginx` 反向代理 `/api`
- 后端支持 Docker 专用配置文件，不影响本地开发使用现有 `config.yaml`
- 上传文件、日志、数据库数据、Redis 数据具备持久化能力
- 提供最小健康检查与明确的测试环境部署文档，便于启动、重建、排障、回滚

## 3. 范围边界

本次只做测试环境最小完整闭环，不扩展到以下范围：

- 不做 Kubernetes、Swarm、Nomad 等编排方案
- 不做域名、HTTPS、证书自动签发
- 不做 CI/CD 自动发布流水线
- 不做正式环境多副本、高可用、灰度发布
- 不接入外部托管 MySQL 或 Redis
- 不改造为运行时动态注入前端环境变量体系

本次方案明确定位为：

- 单机测试环境部署方案
- 仓库内可维护的标准 Docker 资产
- 后续正式环境部署优化的基础版本

## 4. 总体方案

测试环境采用单机 `docker compose` 统一编排四个服务：

- `mysql`
- `redis`
- `server`
- `web`

对外暴露两个访问入口：

- 前端页面：`服务器IP:8081`
- 后端接口与排障入口：`服务器IP:9000`

服务关系如下：

- `mysql` 负责结构初始化与业务数据持久化
- `redis` 负责缓存、鉴权或临时状态支撑
- `server` 连接 `mysql` 和 `redis`，负责 API、静态上传资源、初始化默认数据
- `web` 使用 `nginx` 承载打包产物，并将 `/api/` 请求代理到 `server:9000`

该方案优先保证：

- 一条命令启动
- 一条命令停止
- 可以直接查看容器日志
- 机器 IP 或端口变化时不需要改前端业务代码逻辑

## 5. 配置策略

### 5.1 后端配置

后端当前只读取固定 `config.yaml`。本次需要调整为：

- 默认仍读取 `config.yaml`
- 支持通过启动参数 `-c <config-file>` 指定配置文件

Docker 测试环境新增 `server/config.docker.yaml`，专门用于容器内运行：

- MySQL 主机使用服务名 `mysql`
- Redis 主机使用服务名 `redis`
- 服务监听地址使用容器内可访问方式
- 日志目录保持为容器工作目录下的 `logs`
- Casbin 模型路径继续使用相对路径 `./rbac_model.conf`

这样可以实现：

- 本地开发继续使用原有 `config.yaml`
- Docker 测试环境单独使用 `config.docker.yaml`
- 后续正式环境可以继续扩展新的配置文件，而不破坏已有入口

### 5.2 前端配置

当前 `web/.env.test` 将 `VITE_API_BASE_URL` 写死为外部测试地址。该做法需要调整为：

- 测试环境前端 API 地址统一写为相对路径 `/api`

前端 `vite` 开发代理仍保留现有能力，但测试环境打包产物不再依赖固定 IP。实际 API 目标由 `web` 容器中的 `nginx` 反向代理控制。

这样可以保证：

- 更换测试机 IP 或端口时，不需要重新修改前端接口地址语义
- 浏览器始终访问当前站点同源 `/api`
- 避免前后端跨域配置复杂化

## 6. 容器与镜像设计

### 6.1 后端镜像

后端使用多阶段构建：

- 构建阶段：使用 Go 镜像编译 Linux 可执行文件
- 运行阶段：使用轻量基础镜像，仅保留二进制、配置文件、Casbin 模型文件

镜像内需要包含：

- `server` 可执行文件
- `config.docker.yaml`
- `rbac_model.conf`

运行命令使用：

- `./server -c config.docker.yaml`

### 6.2 前端镜像

前端同样使用多阶段构建：

- 构建阶段：使用 Node 镜像执行依赖安装和 `build:test`
- 运行阶段：使用 `nginx` 提供静态资源与反向代理

镜像内需要包含：

- `dist/` 前端产物
- Docker 测试环境专用 `nginx` 配置

`nginx` 需要实现两件事：

- `history` 路由回退到 `index.html`
- `/api/` 代理到 `http://server:9000`

## 7. 数据初始化策略

测试环境数据库采用“首次启动自动初始化”口径。

初始化来源分两层：

### 7.1 MySQL 容器初始化

`mysql` 容器负责创建数据库实例与基础账号。必要时可通过初始化目录挂载补充建库逻辑，但不依赖整库 SQL 导入作为主路径。

### 7.2 应用启动初始化

后端在启动时继续复用当前已有逻辑：

- `AutoMigrate`
- 默认角色、默认部门、默认用户初始化
- 系统内置配置补齐
- 菜单、API、权限等内置数据补齐

因此首次启动完成后，系统应具备：

- 可访问登录页
- 可调用登录接口
- 可使用默认管理员账号登录

## 8. 持久化设计

测试环境必须对以下数据持久化：

### 8.1 数据卷

- `mysql` 数据目录使用命名卷
- `redis` 数据目录使用命名卷

### 8.2 后端业务目录

后端额外挂载宿主机目录：

- `server/uploads`
- `server/logs`

原因如下：

- `uploads` 保存本地上传文件，容器重建后不能丢失
- `logs` 保存应用日志，便于排查测试环境异常

本次不额外挂载源码目录，不采用开发态热更新模式。

## 9. 健康检查与启动顺序

当前项目没有专门的 Docker 健康检查接口。本次补充最小健康检查能力，例如：

- `GET /health`

该接口只用于：

- 返回服务已启动
- 作为 Docker `healthcheck` 探针入口

`docker compose` 中的健康检查策略应覆盖：

- `mysql`
- `redis`
- `server`

启动关系为：

- `server` 依赖 `mysql`、`redis` 健康后启动
- `web` 依赖 `server` 可用后启动

这里的目的不是构建复杂探活系统，而是避免“容器已启动但服务尚不可用”导致的首轮访问失败。

## 10. 端口规划

测试环境使用固定对外端口：

- `web`：`8081`
- `server`：`9000`
- `mysql`：默认不对公网暴露，必要时仅保留本机映射
- `redis`：默认不对公网暴露，必要时仅保留本机映射

当前优先策略是：

- 页面访问走 `8081`
- 接口直连和联调排障保留 `9000`
- 数据库和 Redis 主要用于容器内互联，避免无意义暴露

## 12. 验证闭环

本次实现完成后，测试环境验证口径固定为：

1. 执行 `docker compose up -d --build`
2. 检查 `mysql`、`redis`、`server`、`web` 容器状态
3. 访问 `http://服务器IP:8081`
4. 调用 `http://服务器IP:9000/health`
5. 使用默认账号 `admin / 123456` 登录后台
6. 验证前端页面请求通过 `/api` 正常转发
7. 验证上传目录与日志目录已在宿主机落盘

通过标准不是“容器能跑”，而是“页面、接口、初始化、代理、持久化”都形成完整闭环。

## 13. 回滚与重建口径

测试环境部署需要明确最小回滚与重建方式：

- 若镜像构建失败或服务异常，可直接 `docker compose down`
- 若需重启保留数据，重新执行 `docker compose up -d --build`
- 若需清空数据库和缓存重新初始化，可执行带卷删除的重建命令

本次不实现复杂版本回滚平台，但文档中必须明确：

- 保留数据重启怎么做
- 清空数据重建怎么做
- 日志查看怎么做

## 14. 最终结论

本次测试环境 Docker 方案正式定义为：

- 使用 `docker compose` 统一编排 `mysql + redis + server + web`
- 前端通过 `nginx` 提供静态资源与 `/api` 反向代理
- 后端通过 `config.docker.yaml` 与 `-c` 参数实现环境隔离
- 首次启动依赖应用内建初始化逻辑自动完成建表与默认数据补齐
- 数据库、缓存、上传文件、日志全部具备持久化能力
- 测试环境访问方式固定为 `服务器IP + 端口`

该方案满足当前封版前“先稳定部署 test 环境”的要求，并为后续正式环境部署优化保留清晰扩展空间。
